# 評審團 — 投資分析品質把關技能

## 技能資訊
- 名稱：評審團
- 版本：1.0
- 類型：品質把關（投資分析的事實檢查與觀點整合）
- 隸屬：推衍系統 / 投資分析模組
- 觸發條件：由投資分析.skill 第六步自動調用，或閣主指令「評審」「審查」「把關」
- 員工來源：Iker 個股研究系統 v4 — 四層框架模組

## 角色定位
評審團是投資分析的品質把關機制，由 5 個專業角色組成。
使用 Iker 四層框架（23 項分析維度）逐層推衍，每項產出觀點。
不設門檻、不排除、逐層推衍 — 缺資料不影響其他層，只通知補資料。

## 核心原則
- **無門檻限制**：不會因某項不通過就排除該標的，每一項都產出觀點
- **逐層推衍**：按第一層→第四層順序，每層每項檢視後產出分析觀點
- **缺資料通知**：推衍時碰到「需手動找」的項目且資料庫查無資料，才通知閣主
- **FIAS 標記**：每個觀點都標註事實等級（F 事實 / I 推論 / A 假設 / S 推測）

## 評審團角色

### 1. 產業分析師 — 負責第一層（產業面篩選 7 項）
分析維度：
1. 供需是否失衡？（供不應求 = 有機會）
2. 失衡原因是什麼？（庫存去化？產能轉移？需求爆發？）
3. 能持續多久？（新產能何時開出？缺口填得完嗎？）
4. 定價權在誰手上？（賣方定價 = 空間大）
5. 合約報價趨勢？（每月追蹤報價走向）
6. 是否主流題材？（市場資金是否關注此產業）
7. 法說會/逐字稿確認？（排除借殼偷轉型）

### 2. 個股研究員 — 負責第二層（個股受惠程度 6 項）
分析維度：
1. 上下游誰更受惠？（這次上游先漲還下游？）
2. 稼動率多少？（低稼動 = 營收還沒反映 = 空間大）
3. 有低價庫存嗎？（報價漲時毛利率會爆發）
4. 月營收趨勢？（放緩+高點 = 準備跑）
5. 上游龍頭有動嗎？（龍頭動 = 產業確認翻好）
6. CB 狀態？（可轉債即將轉換 = 主力有拉抬動機）

### 3. 籌碼分析師 — 負責第三層（籌碼面驗證 5 項）
分析維度：
1. 法人在追嗎？（不計價追 = 極度看好）
2. 主力分點有進嗎？（特定分點連續買入）
3. 分點有換手嗎？（不同分點輪買 = 漲勢延續）
4. 融資比例健康嗎？（50-60% 以下 = 健康）
5. 0050 入選可能？（即將入選有入選行情）

### 4. 策略長 — 負責第四層（空間估算與部位決策 5 項）
分析維度：
1. 歷史類比法（找類似產業循環，折半估算）
2. 法人目標價定位（目標價只是地板）
3. 估值無法算 = 最好時機（沒人算得出 = 不是最後一批）
4. 風險在哪？（供給回頭？需求萎縮？新產能開出？）
5. 部位大小決策（信心度 vs 集中度）

### 5. 風控官 — 橫跨所有層
職責：
- 盤點所有資料缺口，整理成缺口清單
- 標記哪些觀點是「推測」而非「事實」
- 檢查分析中的認知偏誤（確認偏誤、錨定效應、近因偏誤）
- 提出反面觀點和風險警示
- 檢查推論鏈是否完整、有無跳躍

| 第二層 #6 | CB 狀態 | 公開資訊觀測站 |
| 第三層 #1 | 法人在追嗎 | 三大法人買賣超 |

### 需研判（9 項）— AI 推測，標註「推測」
| 位置 | 維度 |
|------|------|
| 第一層 #1 | 供需是否失衡 |
| 第一層 #2 | 失衡原因是什麼 |
| 第一層 #4 | 定價權在誰手上 |
| 第二層 #1 | 上下游誰更受惠 |

| 第四層 #4 | 風險在哪 |
| 第四層 #5 | 部位大小決策 |

### 需手動找（5 項）— 資料庫查不到，通知閣主
| 位置 | 維度 | 建議查找方式 |
|------|------|-------------|
| 第一層 #3 | 能持續多久 | 法說會、產業報告 |
| 第一層 #5 | 合約報價趨勢 | TrendForce、券商產業報告 |
| 第二層 #2 | 稼動率多少 | 法說會逐字稿、券商拜訪報告 |
| 第二層 #3 | 低價庫存 | 財報存貨推估、法說會管理層發言 |

## 執行流程

```
投資分析.skill 第六步呼叫評審團
  │
  ├─ 1. 接收前五步的分析結果（競爭力地圖 + 護城河 + 產業 + 籌碼 + ACH）
  │
  ├─ 2. 逐層推衍（Iker 四層框架）
  │     ├─ 第一層：產業面 7 項 → 產業分析師產出 7 個觀點
  │     ├─ 第二層：個股受惠 6 項 → 個股研究員產出 6 個觀點
  │     ├─ 第三層：籌碼面 5 項 → 籌碼分析師產出 5 個觀點
  │     └─ 第四層：空間估算 5 項 → 策略長產出 5 個觀點
  │
  ├─ 3. 資料缺口處理
  │     ├─ 「可查」→ 引用數據產出觀點
  │     ├─ 「需研判」→ 推測觀點，標註 [S-推測]
  │     └─ 「需手動找」且資料庫無資料 → 通知閣主
  │
  ├─ 4. 風控官橫切檢查
  │     ├─ 標記所有推測項目
  │     ├─ 列出資料缺口清單
  │     ├─ 認知偏誤檢查
  │     └─ 提出反面觀點
  │
  └─ 5. 策略長整合結論
        ├─ 綜合五角色觀點
        ├─ 產出買進/觀望/迴避建議
        ├─ 信心度評估（高/中/低）
        └─ 附註未解決的資料缺口
```

## 通知格式

### 單項缺資料
```
⚠️ 資料缺口通知
─────────────────────────────────
分析維度：稼動率（第二層 #2）
狀態  ：資料庫無相關數據
建議查找：
  - 該公司最近一次法說會逐字稿
  - 券商拜訪報告
  - 產業協會統計數據
─────────────────────────────────
目前以「推測」方式產出觀點，提供資料後可更新。
```

### 多項合併通知
```
⚠️ 資料缺口通知（共 N 項）
─────────────────────────────────
1. 合約報價趨勢（第一層 #5）
   → 建議查找：TrendForce 報告、券商產業報告

2. 稼動率（第二層 #2）
   → 建議查找：法說會逐字稿、券商拜訪報告
─────────────────────────────────
以上項目目前以「推測」方式產出觀點。
提供實際資料後可更新分析結論。
```

## 注意事項
- 評審團不設門檻，不因缺資料而排除分析
- 所有「推測」必須標明，不可偽裝為事實
- 資料缺口通知應合併發送，避免打斷分析流程
- 風控官的反面觀點是強制項，不可省略
- 評審結果歸檔至 藏經閣/投資分析資料/{代號}_{公司名}/第六步_評審團.html
